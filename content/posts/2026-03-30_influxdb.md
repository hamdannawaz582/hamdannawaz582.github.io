# Installing InfluxDB
The latest version of InfluxDB at the time of writing this post is InfluxDB 3 so I'll be going with that (though I might come to regret that decision later). Following the steps given in the [official guide](https://docs.influxdata.com/influxdb3/core/install/), it goes something like this,

### Install via the Quick Installer Script
While running random scripts that do `curl website/script.sh | sh` isn't exactly good practice because you're essentially letting a random script on the network that you don't know the content of run on your system, it's not that big of an issue to me since I'm installing in a fresh LXC, not like I have a pot of gold to protect on my homelab.

**I run the official Quick Install script**
```bash
curl -O https://www.influxdata.com/d/install_influxdb3.sh \
&& sh install_influxdb3.sh 
```

**Here it prompts us for our preferred mode of installation**
![Prompt showing a choice between installing on bare metal or via a docker compose file in a docker container](/images/2026-03-30_influxdb/installation-type-prompt.png)

Now here I chose to install the binary because I prefer running my services in LXCs instead of Docker because of the lower overhead (I'm using proxmox so I have to run Docker in a VM), this ended up being a minor issue for me later on as they don't actually provide a way to install InfluxDB Explorer *without* docker so keep that in mind. Though it shouldn't be too hard to install manually based on the dockerfile, not something I tried so I can't say.

I wish they had made it clearer with what limitations came with JUST the binary because while installing I had no way to know I wouldn't be able to use InfluxDB Explorer. I ended up using Grafana's Explorer later on anyway so no harm done, but it still left a bad taste in my mouth.

**Save the token it generates.**
And with that InfluxDB is installed and ready to go.

# Configuring InfluxDB
There's only really one thing we need to do in InfluxDB right now, create our first bucket.

It took a little bit to wrap my smooth SQL brain around timescale databases but the gist of it is this,
- InfluxDB has buckets which are a bit like SQL databases
- Each bucket has measurements which are like tables (though with a flexible schema)
- In each measurement we use tags to differentiate data emitters and other conditions, e.g 1 tag for pve1 and a separate tag for pve2

There is also one critical difference between InfluxDB buckets and SQL databases, buckets must have a retention period after which time older records are deleted, for example, a bucket with a retention period of 1 hour will only keep the most recent hour's worth of measurements. 

And then I had to throw all that out because InfluxDB 3 has now switched over to SQL-like constructs. Joy. I'll still keep calling it the old way though because I learned those terms and I am damn well gonna use them now.

InfluxDB 3 Core caps us to a max of 3 day retention with the default parameters **at the time of writing**, this limitation is due to it not having the compactor feature of InfluxDB 3 Enterprise (this is the cause for regret I mentioned earlier).

Let's create the main bucket with a retention policy of 1 day,
**Run**
```bash
influxdb3 create database --retention-period 1d --token $INFLUXDB3_AUTH_TOKEN infra_raw
```
Here `infra_raw` is the name of our bucket.

With that our work with InfluxDB is done for the time being. Let's get ourselves some data.

# Connecting Proxmox with InfluxDB
Luckily Promxox makes setting up an InfluxDB metrics server very simple.
- Click on your dataceter
- Scroll down and click on "Metrics Server"
- Click on "Add"
- Click on "InfluxDB"

![Image showing where each button mentioned above is on the Proxmox VE Dashboard](/images/2026-03-30_influxdb/proxmox-metrics-server-location.png)

In the prompt shown, write the following,
- **Name**: Name to show on the "Metric Servers" tab (can be anything)
- **Server**: IP address for your InfluxDB instance (most likely your device's IP if you're using docker, or the container's IP if you're using an LXC like me).
- **Port**: Port of your InfluxDB instance (for me it was 8181, if you're using docker it's whatever you bound it to)
- **Protocol**: HTTP
- **Organization**: Write whatever, it doesn't matter in InfluxDB 3 Core
- **Bucket**: The bucket we made earlier
- **Token**: The token we saved earlier

![Image showing Proxmox's prompt for adding an InfluxDB metric server](/images/2026-03-30_influxdb/proxmox-metric-server-add-prompt.png)

It's important to note that Proxmox mainly has support for InfluxDB V1 & V2 where organizations DO matter, here InfluxDB 3 is using a compatibility layer to facilitate the older connection from Proxmox so I can safely write whatever I want in it.

With this Proxmox's metrics are now being beamed to InfluxDB for us to do with as we please.

# Final Result
With some data finally in InfluxDB, we can now query it to our heart's content, let's see what tables we have,
```bash
influxdb3 query --database infra_raw "SHOW TABLES"
```

```
+---------------+--------------------+------------------------------------------------------------------------------------------------+------------+
| table_catalog | table_schema       | table_name                                                                                     | table_type |
+---------------+--------------------+------------------------------------------------------------------------------------------------+------------+
| public        | iox                | ballooninfo                                                                                    | BASE TABLE |
| public        | iox                | blockstat                                                                                      | BASE TABLE |
| public        | iox                | cpustat                                                                                        | BASE TABLE |
...
| public        | iox                | system                                                                                         | BASE TABLE |
...
```

Pretty cool, eh?

Now the one I'm interested in is `cpustat` which, among other things, houses the CPU usage of our Proxmox nodes,
```bash
influxdb3 query --database infra_raw "SELECT \"cpu\" * 100, \"time\", \"host\" FROM \"cpustat\" WHERE \"host\" = 'pve' ORDER BY \"time\" desc LIMIT 1"
```
Here we pull the CPU utilization (between 0 and 1 for proxmox), time and the host the datapoint is for from the cpustat measurement and get the latest datapoint for the node `pve`.
```
+--------------------------+---------------------+------+
| cpustat.cpu * Int64(100) | time                | host |
+--------------------------+---------------------+------+
| 12.5606331376053         | 2026-03-30T18:51:31 | pve  |
+--------------------------+---------------------+------+
```
As we can see the node is chilling.

One thing you may have noticed throughout this post is, despite me referring to these buckets and measurements and whatnot, InfluxDB actually uses SQL and SQL DB constructs like databases and tables, so what gives?

Well with InfluxDB 3, they have moved away from their own query language to adopt SQL and SQL constructs, I still refer to the "old" names because InfluxDB is still a time series DB and these names are ubiquitous between them, so if I ever need to move over to say VictoriaMetrics, I won't be blindsided like I was when moving from SQL DBs to InfluxDB.
